# Example: Meeting Minutes Output

Below is a real example of well-structured meeting minutes generated from a transcript and raw notes.
Use this as the gold standard for format, tone, and level of synthesis when building the
**4b diarize/meeting** output (Step 4b of `SKILL.md`).

This example is markdown; in `voice-memo-to-notion` you render the same structure as Notion
blocks — the markdown `## N. Topic` → `heading_3`, `**Q. …?**` → a bold-led `paragraph`,
`A. …` → a `bulleted_list_item`, and the **bolded** key terms → `rich_text` segments with
`annotations.bold=true`. What matters is the synthesis depth, the thematic clustering, and the
preserved facts — not the markdown syntax.

# 회의록: 인터뷰이 인터뷰 — OO발전소 현업 워크플로우 파악

**일시:** 2026.03.16
**참석자:** 인터뷰이 (발전 도메인 전문가), 자사 측 (참석자A, 참석자B 등)
**목적:** 발전소 현업 워크플로우 파악 및 AI 적용 지점 구체화

---

## 1. 데이터 시스템 & 인프라

**Q. 발전소에서 주로 사용하는 시스템은?**
A. 크게 두 가지: (1) **도면 시스템** — P&ID, 매뉴얼, 데이터시트, 설계도 등 도면 열람 (2) **PMS** — TM(트러블 메모), 인계 데이터, 설비 정보. 미소(52g의 노코드 AI 플랫폼)가 이 두 시스템을 통합하는 대시보드를 구축하려 하고 있음.

**Q. 도면의 종류는?**
A. 크게 두 유형: (1) **도서** — 매뉴얼, 절차서(줄글), 데이터시트, 자재 스펙 (2) **P&ID** — 배관/설비 계통도. OO발전소은 모든 도면이 한 페이지씩 PDF로 쪼개져 있음. 총 약 4만 페이지, 계통별로 폴더 분류되어 있음.

**Q. 외부에서 API로 데이터를 가져올 수 있는가?**
A. 불가. 국가보안시설이라 외부 API 접근 차단. 엑셀/PDF로 자료를 받아서 별도 환경에 올려 사용해야 함. 자료 수령 자체는 협약 체결로 가능.

**Q. DCS(실시간 모니터링) 데이터 접근은?**
A. DCS 데이터는 RTDB(Real-Time Database)에 내부망 저장. 외부로는 "데이터파크"라는 업체를 통해 단방향 전송 중. 포인트당 비용 지불하고 API 개방 가능. ABB/지멘스 같은 DCS 공급업체가 내부 로직 접근을 차단하고 있어 제어 로직 자체는 가져올 수 없으나, 센서값(디지털/아날로그)은 수신 가능.

**Q. CAD 도면이 있는가?**
A. OO발전소은 건설 당시 계약 미비로 CAD 도면이 거의 없음. P&ID 일부만 CAD 존재 가능성 있음. 타 계열사(다른 계열사)는 OCR 완료 상태.

## 2. 현업 워크플로우 (TM 처리 프로세스)

**Q. TM(트러블 메모)이 올라오면 어떤 과정을 거치나?**
A. TM 접수 → 설비 확인 → 관련 도면 찾기 → 정비 계획 수립. 구체적으로:
1. TM에서 고장 설비 확인 (유니크한 설비명/KKS 코드 포함)
2. 해당 설비의 P&ID 도면을 계통별 폴더에서 탐색
3. 설비 정지 가능 여부, 차단할 밸브/전원 확인
4. 동시간대 근처 다른 작업 유무 확인 (TM 데이터 재참조)
5. 상세 설계도/데이터시트에서 부품 정보 확인
6. 밸브 종류 파악 (전원 차단 시 열림/닫힘 여부) → 준비물 결정

**Q. 도면을 찾는 데 가장 오래 걸리는 부분은?**
A. **P&ID 찾기**가 가장 오래 걸림. P&ID가 텍스트로 긁히지 않아 계통을 따라 폴더를 하나씩 열어봐야 함. 텍스트가 되는 매뉴얼/도서는 Ctrl+F로 KKS 코드 검색 가능하나, P&ID는 수작업 탐색.

**Q. 설비 식별은 어떻게 하나?**
A. 설비마다 유니크한 코드(보통 KKS 코드) 존재. OO발전소은 약간 다른 방식이나 모든 설비에 고유 ID가 있음. TM에도 이 코드가 기재되어 검색 키로 사용. 같은 종류 설비(예: Pulverizer)가 여러 대(A, B, C)면 각각 구분됨.

**Q. 용어 혼재 문제가 있는가?**
A. 심각함. 예: Dust Collector = DC = 백필터 = Bag Filter. 사람마다 표기가 달라 검색을 여러 번 해봐야 함. 유사어 매핑이 안 되어 있음.

**Q. 사람들이 겪고 있는 핵심 문제는?**
A. 봐야 할 파일이 여러 종류(P&ID, 매뉴얼, 데이터시트, TM 이력 등)에 걸쳐 있는데 **한 번에 검색되는 시스템이 없음**. 각각 다른 시스템/폴더에 흩어져 있어 수작업 탐색.

## 3. 이상 감지 & 모니터링

**Q. 이상 감지는 어떻게 이루어지나?**
A. DCS에서 센서값이 실시간 트렌드로 올라오고, High/Low 밴드 초과 시 알람 발생. 운전원이 트렌드 확인 후 현장 운전원에게 점검 지시. 문제는 A 센서 이상 시 연관된 다른 센서도 확인해야 하는데, 계통이 크고 연관 설비가 많아 일일이 확인해야 함.

**Q. 센서 이상이 발생하면?**
A. 수치가 튀었을 때 실제 이상인지 센서 고장인지 확인 필요. 현장 점검 → 과거 이력/트렌드 확인 → 인계일지 확인 → 매뉴얼 참조 → 종합 판단. GPU 모니터링과 유사한 프로세스.

---

## 4. 조직 & 사용자

**Q. 주 사용자는 누구인가?**
A. 감독 부서 (현장 직접 작업X, 공무 역할). 8시간 중 5-6시간 컴퓨터 앞 근무.
- **운전 부서** (7명 교대): 3명 DCS 모니터링 + 4명 현장 운전. 주로 P&ID 참조.
- **정비 부서** (일근직): 디테일한 스펙, 자재 사항, 데이터시트 참조.
- 둘 다 도면 시스템 사용.

**Q. 고장 빈도 통계가 있는가?**
A. 없음. 현재는 주기 단위로 정비 (2년 전에 했으면 안 하는 식). 현업에서는 "최근 1년간 고장 빈도 높은 설비 정비해줘" 같은 분석을 AI에게 시키고 싶어함.

---

## 5. 프로젝트 방향 논의

**Q. 도면을 AI가 해석해서 답변까지 해야 하나?**
A. **파일을 빨리 찾아서 근거와 함께 제시하는 UX가 더 적합**. 판단은 사람에게 맡기는 방식. 도면 해석이 어려운 게 아니라, 연관 도면이 흩어져 있어 찾기가 힘든 것이 핵심 문제.

**Q. 기존 RAG 접근의 한계는?**
A. 미소 플랫폼의 RAG는 빈약. 한 통에 넣고 검색하는 방식으로는 A설비/B설비 유사 고장이 살짝 다를 때 구분 못함. P&ID는 아예 RAG 불가 (텍스트 추출 안 됨). SQL 기반 메타 검색이 오히려 더 유용했음.

**Q. Gemini 테스트 결과는?**
A. OO발전소 P&ID PDF 한 장을 Gemini에 넣고 시뮬레이션했더니 현업 황과장과 거의 유사한 답변. 표준화력이라 학습 데이터에 포함되어 있는 듯. 도면 기호만 인식하면 medium 난이도 태스크.

**Q. 다른 발전소로 확장 가능한가?**
A. OO발전소은 표준화력이라 다른 발전소와 계통 흐름, 기호 명명 규칙이 다를 수 있음. 서치 자체는 문제없으나 도면 해석 시 차이 발생 가능. 프롬프트로 처리 가능한 부분과 학습이 필요한 부분 구분 필요.

**Q. 효과 측정은 어떻게?**
A. 사람이 찾는 시간 vs AI가 찾는 시간 비교. 같은 TM 주제로 테스트. 기존 RAG로 안 되던 것이 agentic 시스템으로 풀리는 것 자체가 성과.

## 6. 기타

**Q. 노하우/경험 의존적 지식은?**
A. 세세한 밸브 차단 순서 등은 경험에 의존. 문서화 안 되어 있음. 프로토타입을 빨리 만들어 현업에게 쓰게 하고 피드백 수집하는 사이클이 필요. 직접 인터뷰도 병행해야 함.

**Q. OCR 현황은?**
A. 과거 비전 모델로 P&ID OCR 시도 → 꽤 괜찮았으나 비용 이슈로 중단. 알바 vs 비전 비교 시 당시에는 알바가 비용 효율적이었음 (1년 이상 전 기준). KKS 코드만 추출하면 될 수도 있음.

**Q. 카카오톡 소통은?**
A. 외주 정비업체는 정비 결과 입력 시스템이 없어 카톡으로 공유. 운전원이 전달 받아 TM에 기록. OO발전소은 비교적 잘 되어 있는 편.
